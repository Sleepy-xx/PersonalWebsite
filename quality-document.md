# PersonalWebsite 质量文档

本文档记录当前 PersonalWebsite 的质量目标、质量标准和发布前检查项。内容基于当前仓库文件观察，不虚构后端、数据库、监控、测试或 CI 能力。

## 项目质量目标

PersonalWebsite 是一个基于 Next.js App Router 的静态个人作品集网站，目标是稳定展示 TSANG KWOK HIN (HENRY) 的个人简介、教育经历、技能和项目作品，并能够通过 GitHub Pages 静态部署公开访问。

当前已发现的主要能力：

- 首页 `/`：个人介绍与技术栈。
- 履历页 `/resume`：教育经历、项目经历和技能。
- 项目列表页 `/project`：读取本地 MDX 项目内容并展示项目卡片。
- 项目详情页 `/project/[slug]`：渲染单个 MDX 项目详情。
- 静态部署：`.github/workflows/deploy.yml` 使用 GitHub Pages 部署 `out/`。

当前未发现相关实现：

- 后端服务、数据库或运行时 API 路由。
- 自动化单元测试、端到端测试或监控告警。
- CMS 或非技术内容管理后台。

## 页面质量标准

- 首页应快速说明站点主人、当前身份、技术方向和作品集入口。
- 履历页应保持教育经历、项目经历、技能列表的时间、名称和描述一致。
- 项目列表页应展示所有正式项目内容，并避免出现空白卡片或缺失图片。
- 项目详情页应能渲染 MDX 内容，且 frontmatter 字段应与卡片和详情页使用方式匹配。
- `About`、`Resume`、`Project` 导航应在桌面和移动端均可访问，并能正确反映当前页面。

待确认：

- `src/app/page.js` 与 `src/app/page.jsx` 同时存在时的实际构建和路由行为。
- 线上 GitHub Pages 子路径下所有图片、链接和详情页路径是否完全正常。

## 内容质量标准

- 个人姓名、身份、学校、联系方式、社交链接应准确，涉及个人隐私的信息应由本人确认后公开。
- 项目内容应避免夸大未实现能力；Demo、Code、Docs 链接如存在，应可访问且与项目匹配。
- `src/content/projects/*.mdx` 的 frontmatter 字段应保持一致，包括 `title`、`category`、`startDate`、`endDate`、`image`、`techStack`、`author` 等当前页面使用的字段。
- 日期、技术名词和英文大小写应统一，减少拼写不一致。
- 未正式接入的 Blog、Gallery 等组件不应在文档中被描述为已上线功能。

## UI/UX 质量标准

- 视觉风格应保持作品集页面的一致暗色主题、卡片层级和强调色。
- 页面信息密度应适合个人网站：简介清楚，履历可扫读，项目详情可深入阅读。
- 链接、按钮和卡片应有明确可点击状态。
- 移动端底部导航不应遮挡主要内容；桌面端侧边栏和内容区应保持可读间距。
- 长文本、长邮箱、项目标题和技术标签不应溢出容器。

## 响应式设计标准

- 至少检查移动端、平板和桌面宽度下的首页、履历页、项目列表页和项目详情页。
- 项目卡片网格应在窄屏下单列显示，在较宽屏幕下保持合理列数。
- 侧边栏在移动端应堆叠在内容上方，在桌面端可保持左侧固定区域。
- 图片应按比例展示，避免拉伸、裁切错误或路径失效。

当前状态：代码中存在响应式 className，但本轮未运行浏览器视口验证，实际表现待确认。

## 可访问性标准

- 根布局语言、页面标题和描述应与站点内容匹配。当前 `layout.js` 使用 `lang="en"` 和基础 metadata。
- 重要图片应提供有意义的 `alt` 文本，装饰性图片应避免干扰阅读。
- 链接文本应能说明目的，例如 GitHub、LinkedIn、Demo、Code、Docs。
- 文本和背景应具备足够对比度，特别是灰色正文、琥珀色强调文本和深色背景。
- 键盘导航、焦点可见性和移动端导航可操作性待确认。

## 性能标准

- 项目应保持静态导出友好，不依赖运行时后端能力。
- 图片资源应控制尺寸和格式，避免无必要的大图拖慢首屏。
- 新增依赖应谨慎评估体积和必要性；当前任务未修改依赖。
- 发布前应至少运行构建命令确认静态导出成功。

当前状态：`next.config.mjs` 已配置 `output: "export"` 和 `images.unoptimized: true`，适配 GitHub Pages 静态部署。实际构建结果待确认。

## SEO 标准

- 站点应有基础 title 和 description，当前已在根布局中发现。
- 关键页面内容应为可静态渲染文本，避免只依赖客户端动态内容。
- 项目详情页应有清楚的标题、分类、日期和正文结构。
- 页面级 metadata、Open Graph、站点地图和 robots 配置当前未发现相关实现，是否需要补充待确认。

## 代码质量标准

- 组件职责应保持清楚：布局、侧边栏、导航、内容区、项目卡片和 MDX 读取逻辑应分离。
- 内容数据来源应可追踪：组件内静态内容和 `src/content/projects/*.mdx` 的边界应清楚。
- 不应在文档中要求当前立刻重构；已知限制应记录为待确认或后续任务。
- 新增代码或内容应通过 ESLint 和构建检查，避免破坏静态导出。
- 当前未发现自动化测试脚本；不要在文档中声称已有测试覆盖。

## 文档质量标准

- README 应说明项目用途、技术栈、本地开发、目录结构和部署方式。
- ARCHITECTURE 应说明路由、组件、数据来源、静态资源和部署流程。
- `feature_list.json` 应记录功能状态、优先级、证据和待确认事项。
- `init.sh` 应作为安全的启动辅助脚本，避免默认安装依赖、提交、推送、删除文件或覆盖配置。
- 本质量文档和评估标准应随项目状态更新，尤其是新增页面、部署变化或验证规则变化时。

## 发布前检查清单

- 确认没有无关业务代码、配置或依赖变更。
- 运行 `npm run lint`，记录结果；如未运行，说明原因。
- 运行 `npm run build`，确认静态导出成功；如未运行，说明原因。
- 本地或线上打开 `/`、`/resume`、`/project` 和至少一个 `/project/[slug]`。
- 检查移动端和桌面端导航、布局、图片和链接。
- 检查 GitHub Pages 工作流是否仍指向 `out/` 静态产物。
- 更新 `feature_list.json` 或进度文档中的验证证据。
